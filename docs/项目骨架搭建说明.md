# 项目骨架搭建说明

## 1. 这次我做了什么

本次不是直接进入功能开发，而是先把项目整理成一个能持续迭代的工程骨架，主要完成了以下内容：

1. 建立单仓结构，拆分为前端 `apps/web`、后端 `apps/api`、文档 `docs`、存储预留目录 `storage`
2. 为前端放入 React + TypeScript + Vite 的最小入口页面
3. 为后端放入 FastAPI 的最小 API 服务和基础路由
4. 预留论文、任务等核心领域对象的 schema 边界
5. 补充根目录 `README.md` 和本说明文档，明确当前骨架的目标与后续扩展方式

## 2. 为什么这样搭

### 2.1 先收敛 MVP，不做“大而全”

根据需求分析文档，当前最重要的不是一次性把解析、翻译、问答、综述全部做完，而是先保证这条闭环能成立：

1. 有地方上传和管理论文
2. 有地方展示原始摘要与论文总结
3. 有翻译能力的清晰落点
4. 有可以继续接入问答、对比、综述的后端边界

所以这次骨架的重点是“把系统边界搭对”，而不是“先堆很多页面或很多接口”。

### 2.2 为什么用前后端分离

这个项目不是纯静态展示站，后续一定会涉及：

- PDF 文件上传
- 论文解析
- 元数据抽取
- 论文总结与翻译任务
- 基于原文的问答和证据返回
- 对比与综述的异步处理

这些能力天然要求后端存在。如果现在用单体前端硬撑，后面一定重构。  
所以第一步直接定成：

- 前端负责交互、任务状态展示和结果呈现
- 后端负责文件、任务、AI 编排和数据接口

这是更稳妥的项目经理式取舍。

### 2.3 为什么后端选 FastAPI

选 FastAPI 不是为了追热门，而是因为它更符合当前任务：

- Python 生态对 PDF 解析、OCR、NLP、翻译和大模型接入更成熟
- 后续如果接 `pdfplumber`、`pymupdf`、`unstructured`、`paddleocr` 或模型 SDK，路径更短
- Pydantic 对数据结构定义清晰，适合把论文、任务、翻译结果这些对象先建模清楚
- 对个人项目而言，开发速度高，调试成本低

另外，远程仓库原本的 `.gitignore` 就明显偏 Python，这和后端选择也是一致的。

### 2.4 为什么前端选 React + TypeScript + Vite

需求文档里后续会有这些页面：

- 论文列表
- 论文详情
- 原摘要 / 总结 / 翻译切换
- 对比结果页
- 综述任务页
- 问答页

这些页面都属于典型的状态型界面，React 更适合长期演进。  
TypeScript 的价值在于先把接口类型收敛，避免前后端联动时边界失控。  
Vite 的价值则是轻量，适合个人项目快速启动。

### 2.5 为什么是单仓 monorepo 结构

当前仓库规模不大，但前后端一定会同时演进。把它们放在一个仓库里有几个现实好处：

- 一次提交可以同时修改前端页面、后端接口和文档
- 产品边界和技术边界能一起被版本管理
- 对课程项目或答辩展示更友好，仓库结构一眼能看懂

所以不是为了“追求 monorepo”，而是因为这个项目现在就适合单仓管理。

## 3. 当前目录为什么这样设计

```text
DocMind/
  apps/
    api/
    web/
  docs/
  storage/
```

### `apps/api`

后端服务入口。这里先定义 API、schema 和基础配置，后续再逐步增加：

- `services/`：论文解析、翻译、总结、问答编排
- `repositories/`：数据库访问
- `workers/`：异步任务执行

这次我没有一次性把这些全建出来，是因为空目录太多会变成噪音。先保留最小可用结构，更利于后续真实生长。

### `apps/web`

前端展示入口。当前页面故意没有假装做成完整产品，而是明确告诉你：

- 当前是 MVP 骨架
- 核心模块是什么
- 初始 API 规划是什么

这比做一个花哨但没有真实边界的首页更有工程价值。

### `docs`

文档不能继续只停留在需求分析。  
从项目管理角度，必须把“为什么这样搭”单独写出来，否则后面每做一步都会重新争论目录和技术路线。

### `storage`

这个目录是提前为真实论文文件、解析中间结果、任务产物预留的。  
原因很简单：DocMind 不是纯数据库系统，它一定有文件资产。

## 4. 这次没有做什么，为什么没做

### 4.1 没有直接接数据库

原因：当前首要任务是把模块边界搭起来，不是过早选 SQLite 还是 PostgreSQL。  
后续可以先用 SQLite 跑通 MVP，再平滑切 PostgreSQL。

### 4.2 没有直接接大模型或翻译 SDK

原因：这一步如果现在就硬接，会把项目骨架和具体供应商绑定在一起。  
现阶段更重要的是先把任务入口和数据结构定义出来，后面接 OpenAI、阿里云还是本地模型都能替换。

### 4.3 没有直接实现上传 PDF

原因：上传只是接口的一部分，真正复杂的是后面的解析、校正和任务编排。  
所以这次先把后端 API 服务、schema 和任务视角搭好，再进文件流处理。

## 5. 我对下一阶段的建议

按项目经理视角，下一步不要同时铺开所有模块，建议按下面顺序推进：

1. 后端先接入 `papers` 的创建、详情、内存存储或 SQLite 存储
2. 前端做论文列表页和论文详情页
3. 打通“原始摘要 + 论文总结 + 翻译状态”的展示链路
4. 再接入文件上传与解析
5. 最后引入任务队列或异步任务模型

这个顺序的好处是：

- 每一步都可演示
- 每一步都能验证需求文档里的核心价值
- 不会因为 OCR、向量检索、模型接入把项目一开始就拖重

## 6. 当前骨架如何判断算是“搭对了”

我用下面四个标准判断这次搭建是合理的：

1. 目录一眼能看懂产品边界
2. 前后端都已有最小可运行入口
3. 论文、任务这些核心对象已经开始建模
4. 文档能解释结构，而不是只留下代码

这四点满足后，项目就从“只有需求文档”进入了“可持续开发”的状态。
