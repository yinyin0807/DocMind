# DocMind 前后端字段对齐清单

## 1. 文档目标

本文档用于把当前前端页面中已经实际使用的数据字段，与后端未来需要提供的接口字段进行一一对应。

它的目标是：

- 避免后端只按理想模型设计，脱离前端真实需求；
- 避免前端继续长期依赖本地 mock 数据；
- 为后续前后端联调提供清晰边界；
- 让后端开发优先服务当前已经做出来的主工作台。

当前主要参考对象：

- `apps/web/src/App.tsx`
- 《后端收敛方案》
- 《后端表结构与 API 草案》

## 2. 当前前端页面结构回顾

当前前端页面已经明确分为两部分：

1. 欢迎页
2. 主功能页

其中真正需要后端支撑的是主功能页。

主功能页目前分为三块：

- 左侧论文列表区
- 右侧当前论文内容区
- 底部对话区

因此，后端第一阶段也应严格围绕这三块提供数据。

## 3. 前端当前核心数据对象

当前前端在 `App.tsx` 中使用了 `Paper` 类型作为主工作台的数据模型。

字段如下：

- `id`
- `title`
- `authors`
- `year`
- `venue`
- `tags`
- `status`
- `note`
- `focus`
- `abstract`
- `summaryShort`
- `summaryLong`
- `researchProblem`
- `method`
- `dataset`
- `results`
- `innovation`
- `limitations`
- `translationTitle`
- `translationAbstract`
- `translationSummary`
- `citations`
- `records`
- `quickQuestions`
- `messages`

注意：

这个 `Paper` 类型当前是为了前端原型方便而做的聚合结构，但后端不应该机械照抄成一张表。
后端应把这些字段拆解到：

- `papers`
- `paper_analyses`
- `chat_sessions`
- `chat_messages`
- `tasks`

## 4. 左侧论文列表区字段对齐

左侧论文列表当前展示内容包括：

- 论文标题
- 作者
- 年份
- 状态
- 标签
- 简短备注

### 前端字段

| 前端字段 | 用途 |
|---|---|
| `id` | 作为选中和切换的主键 |
| `title` | 论文标题 |
| `authors` | 作者展示 |
| `year` | 年份展示 |
| `status` | 列表状态标识 |
| `tags` | 标签展示 |
| `note` | 辅助说明 |

### 后端接口建议

由以下接口提供：

```http
GET /api/papers
```

### 建议响应字段

| 后端字段 | 前端字段 | 说明 |
|---|---|---|
| `id` | `id` | 主键 |
| `title` | `title` | 标题 |
| `authors` | `authors` | 作者文本 |
| `year` | `year` | 年份 |
| `status` | `status` | 前端显示状态 |
| `tags` | `tags` | 字符串数组 |
| `note` | `note` | 简要备注 |
| `venue` | 可选 | 当前左侧未直接使用，但后续可能展示 |

### 说明

- `status` 和 `note` 第一阶段允许后端直接面向 UI 输出；
- `tags` 第一阶段可先由简单字段或拼装结果返回；
- 只要这个接口先稳定，左侧列表就能先从 mock 切成真实数据。

#### 前端字段

| 前端字段 | 用途 |
|---|---|

| `translationAbstract` | 摘要翻译 |

| 前端字段 | 用途 |

```ts

```

建议后端先返回结构化对象：

```json
{
  "citations": {
    "gbt": "...",
    "apa": "...",
    "ieee": "...",
    "mla": "..."
  }
}
```

前端可映射成当前数组结构。

### 6.6 记录标签

#### 前端字段

| 前端字段 | 用途 |
|---|---|
| `records` | 展示操作时间线 |

第一阶段可以直接返回字符串数组：

```json
{
  "records": [
    "2026-03-10 20:10 生成第一版结构化总结",
    "2026-03-10 20:24 生成中译文摘要"

```

## 7. 底部对话区字段对齐

底部对话区当前使用字段包括：

- `messages`
- `quickQuestions`
- `draftQuestion`

其中需要后端真实支持的是 `messages` 和提问提交。

### 7.1 消息历史

#### 前端字段

当前前端消息结构：

```ts
messages: {
  role: "user" | "assistant";
  content: string;
  timestamp: string;
}[]
```

#### 后端接口建议

```http
GET /api/papers/{paper_id}/chat
```

#### 建议响应字段

| 后端字段 | 前端字段 | 说明 |
|---|---|---|
| `messages[].role` | `messages[].role` | 消息角色 |
| `messages[].content` | `messages[].content` | 消息内容 |
| `messages[].created_at` | `messages[].timestamp` | 前端可做格式化 |
| `messages[].evidence` | 暂未使用 | 后续证据定位 |

### 7.2 提交问题

#### 前端行为

当前前端在输入框中输入问题，并点击发送按钮。

#### 后端接口建议

```http
POST /api/papers/{paper_id}/chat
```

#### 建议请求体

```json
{
  "question": "这篇论文的核心贡献是什么？"
}
```

#### 建议响应体

```json
{
  "user_message": {
    "role": "user",
    "content": "这篇论文的核心贡献是什么？",
    "created_at": "2026-03-12T21:00:00+08:00"
  },
  "assistant_message": {
    "role": "assistant",
    "content": "核心贡献在于把引文图结构引入检索表示学习。",
    "created_at": "2026-03-12T21:00:01+08:00",
    "evidence": []
  }
}
```

### 7.3 快捷提问

#### 当前情况

前端 `quickQuestions` 目前是 mock 数据的一部分。

#### 建议

第一阶段可以继续由前端保留，不需要后端返回。

原因：

- 它本质上是 UI 引导，不是核心业务数据；
- 后端先不要承担不必要的展示职责；
- 等问答能力稳定后，再考虑是否按论文类型动态生成快捷问题。

## 10. 第一阶段允许继续 mock 的字段

以下字段第一阶段可以不由后端提供：

- `quickQuestions`
- 更复杂的任务进度细节
- 证据片段 `evidence`
- 多会话切换能力
- 动态推荐问题

## 11. 对后端实现的直接指导意义

基于当前前端，后端开发时应遵循以下优先级：

1. 先保证字段名和结构能让前端容易映射
2. 先保证列表、详情、分析、消息历史能返回稳定数据
3. 不急于做完美抽象，但不要破坏未来扩展空间
4. 不要让前端为了接接口而大改主工作台结构

## 12. 结论

当前后端不能只参考需求文档，还必须直接参考前端已经写出来的工作台。

最关键的事实是：

- 前端已经定义了主交互结构；
- 后端应服务这个结构，而不是反过来让前端迁就后端；
- 第一阶段最重要的是把前端的 `Paper` 聚合数据拆回合理的后端对象，并通过接口重新提供给前端。

因此，后续正式写后端代码时，应优先以这份字段对齐清单作为实现参考。
