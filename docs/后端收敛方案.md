# DocMind 后端收敛方案

## 1. 文档目标

本文档用于收敛 DocMind 当前阶段的后端设计方案，目标不是一次性铺满所有能力，而是先围绕已经完成的前端主工作台，建立稳定、可扩展、可迭代的后端边界。

当前约束如下：

- 先服务现有前端页面结构；
- 先围绕单篇论文工作流搭建后端；
- 先收敛数据模型和 API 设计；
- 暂不优先接入复杂模型能力和复杂任务基础设施；
- 为后续 PDF 解析、总结、翻译、问答、对比、综述预留扩展空间。

## 2. 当前阶段的后端目标

当前阶段后端只解决以下问题：

- 给前端主功能页提供真实数据来源；
- 围绕“单篇论文”建立完整的数据结构；
- 让左侧论文列表、右侧论文详情、底部问答区有稳定接口；
- 为后续异步任务能力保留统一任务模型。

换句话说，当前不是先做“全能科研助手”，而是先做“能支撑主工作台的论文后端”。

## 3. 总体设计原则

### 3.1 以论文为核心对象

系统中的主要业务对象不是聊天，而是论文本身。

因此后端设计必须围绕论文展开：

- 论文列表来自 `Paper`
- 单篇论文分析结果来自 `PaperAnalysis`
- 围绕论文的问答历史来自 `ChatSession / ChatMessage`
- 总结、翻译、对比、综述等过程通过 `Task` 承载

### 3.2 先服务单篇论文工作流

当前前端已经明确采用：

- 左侧论文列表
- 右侧当前论文工作区
- 底部对话区

因此后端第一阶段也必须围绕这个结构来收敛，而不是先做多论文对比或综述生成。

### 3.3 高风险能力任务化

总结、翻译、对比、综述等能力具有以下特点：

- 耗时较长；
- 输出具有不确定性；
- 后续很可能接入模型服务；
- 需要保留任务状态与结果记录。

因此这些能力建议从设计上就纳入 `Task` 模型，而不是全部做成同步接口。

### 3.4 数据与业务逻辑分层

后端不应把数据库操作、业务逻辑和接口路由全部写在一起。

建议坚持：

- API 只负责接收和返回；
- Service 负责业务编排；
- Repository 负责数据访问；
- Model / Schema 负责结构定义。

## 4. 后端模块划分

建议将后端拆分为以下层次：

1. `api`
负责路由定义、请求入口与响应出口。

2. `schemas`
负责 Pydantic 请求和响应模型。

3. `models`
负责数据库实体定义。

4. `services`
负责业务逻辑编排。

5. `repositories`
负责数据库访问，隔离 ORM 细节。

6. `tasks` 或 `workers`
负责异步任务执行，第一阶段可以先占位。

## 5. 目录结构建议

建议目录如下：

```text
apps/api/app/
  api/
    routes/
      health.py
      papers.py
      analyses.py
      chats.py
      tasks.py
  core/
    config.py
    db.py
  models/
    paper.py
    paper_analysis.py
    chat.py
    task.py
  repositories/
    paper_repository.py
    analysis_repository.py
    chat_repository.py
    task_repository.py
  schemas/
    paper.py
    analysis.py
    chat.py
    task.py
  services/
    paper_service.py
    analysis_service.py
    chat_service.py
    task_service.py
  main.py
```

这套目录结构的目的不是“看起来完整”，而是为了避免后续代码全部堆到 route 文件里。

## 6. 核心领域对象

第一阶段建议只保留四个核心对象：

### 6.1 `Paper`

论文主对象，承载论文本体与基础元数据。

### 6.2 `PaperAnalysis`

单篇论文分析结果，承载：

- 结构化总结
- 翻译结果
- 引文结果
- 与单篇论文分析相关的衍生内容

### 6.3 `ChatSession / ChatMessage`

围绕单篇论文的问答记录。

### 6.4 `Task`

统一任务对象，用于承载：

- 总结任务
- 翻译任务
- 后续对比任务
- 后续综述任务

## 7. 数据表收敛方案

## 7.1 `papers`

字段建议：

- `id`
- `title`
- `authors`
- `year`
- `venue`
- `doi`
- `abstract_raw`
- `full_text`
- `language`
- `parse_status`
- `is_favorite`
- `reading_status`
- `created_at`
- `updated_at`

用途说明：

- 为左侧论文列表提供数据；
- 为右侧论文头部和元数据区提供基础字段；
- 作为后续总结、翻译、问答的主关联对象。

## 7.2 `paper_analyses`

字段建议：

- `id`
- `paper_id`
- `summary_short`
- `summary_long`
- `research_problem`
- `method`
- `dataset`
- `results`
- `innovation`
- `limitations`
- `title_translation_zh`
- `abstract_translation_zh`
- `summary_translation_zh`
- `citation_gbt`
- `citation_apa`
- `citation_ieee`
- `citation_mla`
- `version`
- `created_at`
- `updated_at`

用途说明：

- 为右侧“概览”标签提供结构化字段；
- 为“AI 总结”“翻译”“引文”标签提供数据；
- 支持后续保留多版本分析记录。

## 7.3 `chat_sessions`

字段建议：

- `id`
- `paper_id`
- `title`
- `created_at`
- `updated_at`

用途说明：

- 用于按论文管理对话会话；
- 为后续“多个对话线程”保留扩展空间。

## 7.4 `chat_messages`

字段建议：

- `id`
- `session_id`
- `role`
- `content`
- `evidence_json`
- `created_at`

用途说明：

- 为底部问答区提供历史消息；
- 后续可扩展证据片段、页码、来源段落等信息。

## 7.5 `tasks`

字段建议：

- `id`
- `task_type`
- `target_type`
- `target_id`
- `status`
- `input_payload`
- `result_payload`
- `error_message`
- `created_at`
- `updated_at`

用途说明：

- 用统一模型管理耗时任务；
- 支持后续轮询状态、查看结果、失败重试；
- 不让总结、翻译、综述等能力直接绑死在同步接口上。

## 8. 数据建模原则

### 8.1 不把所有内容塞进 `Paper`

`Paper` 应只承载论文本体和元数据。

不建议把以下内容直接塞入 `Paper`：

- 详细总结
- 翻译结果
- 问答记录
- 任务结果

否则后续扩展会很快变得混乱。

### 8.2 `PaperAnalysis` 只承载单篇分析结果

该表的定位是“单篇论文的结构化分析缓存和产出”，而不是论文本身。

### 8.3 `Task` 是统一的异步边界

即使第一阶段暂时没有真正的异步执行器，也建议保留统一 `Task` 模型。

这样后续从同步过渡到异步时，前后端接口不需要推翻重来。

## 9. 第一阶段 API 收敛方案

当前建议只实现与现有前端直接对应的接口。

## 9.1 `GET /api/papers`

用途：

- 驱动左侧论文列表。

建议返回字段：

- `id`
- `title`
- `authors`
- `year`
- `venue`
- `tags`
- `status`
- `note`

## 9.2 `GET /api/papers/{paper_id}`

用途：

- 驱动右侧论文头部与元数据区。

建议返回字段：

- 标题
- 作者
- 年份
- 来源
- 研究焦点
- 标签
- 阅读状态
- 收藏状态

## 9.3 `GET /api/papers/{paper_id}/analysis`

用途：

- 驱动右侧标签区的主要内容。

建议返回字段：

- 概览字段
- 原始摘要
- 简洁总结
- 详细总结
- 翻译结果
- 引文结果
- 相关记录摘要

## 9.4 `GET /api/papers/{paper_id}/chat`

用途：

- 驱动底部问答历史区域。

建议返回内容：

- `session`
- `messages`

## 9.5 `POST /api/papers/{paper_id}/chat`

用途：

- 提交当前论文下的一个问题。

第一阶段建议：

- 可以先返回 mock 回答；
- 或先把消息写入数据库，再返回简单占位回答；
- 暂时不急着接真实检索与模型调用。

## 9.6 `POST /api/papers/{paper_id}/summary-tasks`

用途：

- 创建单篇论文总结任务。

第一阶段建议：

- 先把任务建出来；
- 即使暂时不真正执行，也先把任务模型、状态字段和结果结构定下来。

## 9.7 `POST /api/papers/{paper_id}/translation-tasks`

用途：

- 创建翻译任务。

## 9.8 `GET /api/tasks/{task_id}`

用途：

- 查询任务状态与结果。

## 10. 当前阶段不优先实现的内容

以下内容建议暂缓，不进入第一阶段实现：

- 多论文对比真实生成逻辑
- 文献综述生成逻辑
- PDF 上传解析全链路
- 向量检索
- 证据定位
- 用户系统
- 权限系统
- 复杂异步任务基础设施
- 多租户设计

原因很简单：

当前真正需要的是先把单篇论文工作流跑通，而不是提前把所有科研辅助能力堆满。

## 11. 技术选型建议

第一阶段建议采用：

- `FastAPI`
- `Pydantic`
- `SQLAlchemy 2.x`
- `SQLite`
- `Alembic`

原因：

- 足够稳定；
- 足够常规；
- 学习资料和文档多；
- 对个人项目的复杂度更合适；
- 后续迁移 PostgreSQL 成本较低。

当前不建议第一版就引入：

- `Celery`
- `Redis`
- `PostgreSQL`
- 向量数据库
- 复杂消息队列

## 12. 与前端页面的对应关系

当前前端页面已经明确为：

- 左侧论文列表
- 右侧当前论文详情与标签区
- 底部对话区

因此后端第一阶段必须优先支撑以下链路：

1. `/api/papers`
用于左侧列表

2. `/api/papers/{id}`
用于右侧头部和元信息

3. `/api/papers/{id}/analysis`
用于右侧概览、摘要、总结、翻译、引文

4. `/api/papers/{id}/chat`
用于底部消息流

如果这四条链路先跑通，当前前端就可以从 mock 数据切到真实后端数据。

## 13. 推荐开发顺序

建议按以下顺序推进：

1. 先建立数据库连接与基础模型
2. 实现 `papers` 列表接口
3. 实现 `paper detail` 详情接口
4. 实现 `analysis` 接口
5. 实现 `chat` 读取接口
6. 实现 `chat` 写入接口
7. 最后补 `task` 占位接口

这个顺序的核心逻辑是：

- 先让页面可读；
- 再让页面可写；
- 最后为复杂能力补任务边界。

## 14. 当前阶段的核心取舍

这版方案的核心取舍是：

- 优先单篇论文，不优先多论文任务；
- 优先数据结构稳定，不优先模型能力完整；
- 优先服务现有前端，不优先铺满未来功能；
- 优先 SQLite 快速落地，不优先复杂基础设施。

## 15. 结论

当前最适合 DocMind 的后端收敛方案是：

- 以 `Paper + PaperAnalysis + Chat + Task` 作为四个核心对象；
- 以单篇论文工作流作为第一阶段主链路；
- 以 `SQLite + FastAPI + SQLAlchemy + Alembic` 作为当前阶段技术方案；
- 先打通前端主工作台所依赖的论文列表、详情、分析、问答接口；
- 后续再逐步接入 PDF 解析、模型能力、对比分析和综述生成。

如果这条链路先跑通，DocMind 就会从“前端原型”进入“可真实交互的产品雏形”。
