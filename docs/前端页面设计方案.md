# DocMind 前端页面设计方案

## 1. 文档目标

本文档用于定义 DocMind 第一阶段前端页面方案，供你查看、讨论，并作为后续前端实现的直接依据。

当前约束如下：

- 暂不连接后端；
- 先用静态页面和 mock 数据完成前端原型；
- 第一阶段只聚焦两个页面：欢迎页、主功能页；
- 主功能页承载核心功能，不把高频操作拆散到多个页面；
- 用户围绕“当前选中的论文”完成阅读、总结、翻译、问答等操作。

## 2. 设计原则

### 2.1 产品定位原则

DocMind 不是通用聊天工具，也不是普通文件管理器，而是面向科研人员的论文管理与智能分析平台。

### 2.2 页面设计原则

- 以论文为中心，而不是以聊天为中心；
- 以工作流为中心，而不是以功能堆叠为中心；
- 保留学术感、秩序感和可信度；
- 支持后续平滑接入对比、综述、引文等能力；
- 即使在没有后端时，也应具备完整演示价值。

### 2.3 第一阶段体验目标

- 用户一进入系统，就能理解平台是做什么的；
- 用户进入主功能页后，不需要频繁跳转；
- 用户点击不同论文时，右侧工作区立即切换；
- 用户可以直接在论文上下文中对话式完成总结、提问等操作。

## 3. 页面范围

第一阶段只做两个页面：

1. 欢迎页
2. 主功能页

页面关系如下：

```text
欢迎页
  -> 主功能页
```

## 4. 核心工作流

```text
进入欢迎页
  -> 理解产品价值与主要能力
  -> 点击进入主功能页
  -> 在左侧选择一篇论文
  -> 右侧切换为该论文内容
  -> 查看摘要、总结、翻译、引文等信息
  -> 通过底部或右侧对话框围绕该论文提问
  -> 切换另一篇论文继续工作
```

## 5. 页面一：欢迎页

## 5.1 页面定位

欢迎页承担“建立产品认知 + 引导进入工作台”的作用。

它不承担具体业务操作，不做复杂表单，不强调营销，而是帮助用户迅速理解：

- 平台的核心价值；
- 平台区别于普通聊天工具或文献管理工具的地方；
- 进入系统后主要能做什么。

## 5.2 页面结构

欢迎页建议包含以下五个区块。

### A. 顶部导航

- 左侧：DocMind 标识
- 右侧：
  - 产品说明
  - 功能概览
  - 进入工作台

### B. Hero 首屏

建议内容：

- 主标题：论文管理与智能分析平台
- 副标题：围绕论文完成管理、理解、总结、翻译与问答
- 主要按钮：
  - 进入工作台
  - 查看核心功能
- 配图：主工作台示意图或抽象信息结构图

首屏文案应强调三件事：

- 保留原文依据；
- 支持结构化理解；
- 面向真实科研工作流。

### C. 核心价值区

建议用三张卡片表达：

1. 原文与 AI 结果并存
2. 论文库变为可组织、可分析的知识资产
3. 在同一工作台中完成阅读、总结与问答

### D. 核心功能区

建议以网格卡片方式展示：

- 论文导入与管理
- 单篇论文总结
- 原始摘要对照
- 中英翻译
- 对话式问答
- 多论文对比与综述
- 引文生成

### E. 底部行动区

- 简短收束文案
- 按钮：进入主功能页

## 5.3 欢迎页交互要求

- 点击“进入工作台”跳转到主功能页；
- 点击“查看核心功能”滚动到功能区；
- 不加入需要后端支持的功能；
- 首屏按钮位置要足够突出，避免用户进入后不知如何开始。

## 6. 页面二：主功能页

## 6.1 页面定位

主功能页是系统的核心工作台。

用户进入该页面后，应能直接完成以下行为：

- 查看论文列表；
- 搜索和筛选论文；
- 选择一篇论文；
- 查看该篇论文的原始摘要、AI 总结、翻译和引文；
- 围绕当前论文进行对话式提问；
- 为后续扩展多论文对比、综述任务预留入口。

结论是：主功能页不是单纯的论文列表页，而是单页式工作台。

## 6.2 总体布局

建议采用“两区主结构 + 顶部工具栏”的布局，而不是再拆更多独立页面。

```text
顶部工具栏
┌─────────────────┬────────────────────────────────────┐
│ 左侧：论文列表区 │ 右侧：当前论文主工作区              │
│                 │                                    │
│ 搜索 / 筛选      │ 论文头部信息                        │
│ 论文列表         │ 内容切换区                          │
│                 │ 对话区                              │
└─────────────────┴────────────────────────────────────┘
```

## 6.3 顶部工具栏

建议包含：

- 左侧：DocMind / Workspace
- 中部：全局搜索占位或快捷操作入口
- 右侧：
  - 导入论文按钮
  - 最近任务按钮
  - 设置入口

第一阶段可以只做静态按钮，不做实际功能。

## 6.4 左侧论文列表区

## 6.4.1 功能定位

左侧区域用于承载论文集合，是用户进入单篇论文上下文的入口。

用户点击左侧某篇论文后，右侧主工作区整体切换到该论文。

## 6.4.2 结构建议

从上到下依次为：

### A. 标题与统计

- 标题：我的论文
- 辅助信息：总论文数、已读数、收藏数

### B. 搜索框

支持本地筛选：

- 按标题搜索
- 按作者搜索
- 按关键词搜索

### C. 快捷筛选区

建议先做以下标签：

- 全部
- 最近阅读
- 收藏
- 待精读
- 已总结

### D. 论文列表

每个列表项建议展示：

- 论文标题
- 作者或第一作者
- 年份
- 状态标签
- 简短说明，例如“已生成总结”“有问答记录”

当前被选中的论文需要明显高亮。

## 6.4.3 交互规则

- 点击某篇论文后，右侧整体切换到该论文内容；
- 默认打开该论文的“概览”内容；
- 若该论文已有历史消息，对话区同步切换；
- 左侧搜索和筛选只改变列表展示，不中断右侧工作内容；
- 收藏、删除、标记已读等操作先做静态按钮或假交互。

## 6.5 右侧主工作区

## 6.5.1 设计目标

右侧区域必须围绕“当前选中的一篇论文”展开。

也就是说，问答、总结、翻译、引文都不是独立页面，而是当前论文的工作标签。

这符合科研阅读真实流程：先选论文，再围绕论文处理问题。

## 6.5.2 右侧整体分层

建议分为三层：

### 第一层：论文头部信息

显示当前论文基础信息：

- 标题
- 作者
- 年份
- 来源
- 关键词
- 标签
- 阅读状态

同时放置快捷操作按钮：

- 生成总结
- 翻译
- 复制引文
- 加入对比
- 更多操作

### 第二层：内容切换区

这一层负责“点击后右侧内容随之改变”的主要交互。

建议做成标签页或次级导航，包含：

1. 概览
2. 原始摘要
3. AI 总结
4. 翻译
5. 引文
6. 记录

说明：

- 当前阶段先聚焦单篇论文能力；
- 多论文对比、综述任务先保留为入口或占位；
- 不建议第一阶段拆成多个独立详情页。

### 第三层：对话区

对话区固定存在于右侧下方，始终绑定当前论文。

用户在任意标签下都可以继续提问，不需要跳到单独聊天页。

这也是主功能页一体化设计的关键。

## 6.6 右侧内容切换区详细设计

## 6.6.1 概览

用于快速了解一篇论文的核心信息。

建议用结构化卡片展示：

- 一句话总结
- 研究问题
- 核心方法
- 数据与实验
- 主要结果
- 创新点
- 局限性

概览区域应适合快速扫读，不追求长文本。

## 6.6.2 原始摘要

用于展示论文原始摘要。

要求：

- 明确标注“原始摘要”；
- 保留原始内容感，不做 AI 改写伪装；
- 提供复制按钮；
- 后续可扩展术语高亮和段落定位。

## 6.6.3 AI 总结

用于展示系统归纳后的论文总结。

建议包含：

- 简洁版总结
- 详细版总结
- 结构化字段拆分

要求：

- 明确标注为系统生成；
- 后续支持重新生成；
- 后续支持“简洁 / 完整”切换。

## 6.6.4 翻译

用于展示标题、摘要、总结的翻译结果。

建议结构：

- 标题翻译
- 摘要翻译
- 总结翻译
- 原文 / 译文切换

要求：

- 翻译不能覆盖原文；
- 适合后续支持中译英、英译中切换；
- 当前阶段可先用 mock 数据展示。

## 6.6.5 引文

用于展示当前论文的引文结果。

建议至少包含：

- GB/T 7714
- APA
- IEEE
- MLA

每种格式配一个复制按钮。

## 6.6.6 记录

用于展示当前论文的操作沉淀。

建议包含：

- 最近一次总结时间
- 最近一次翻译时间
- 最近几条问答记录
- 是否加入过对比任务

第一阶段可以用时间线和空状态展示。

## 6.7 对话区设计

## 6.7.1 对话区定位

对话区不是一个独立 AI 聊天产品，而是当前论文的问答面板。

用户问的每一句话，都默认针对当前选中的论文。

## 6.7.2 对话区结构

建议包含：

- 标题：针对当前论文提问
- 消息列表
- 输入框
- 发送按钮
- 快捷提问按钮

快捷提问建议：

- 这篇论文的核心贡献是什么？
- 它使用了什么数据集？
- 它的局限性有哪些？
- 能否概括它的研究方法？

## 6.7.3 对话区交互规则

- 用户切换论文时，对话历史同步切换；
- 每篇论文有自己独立的对话上下文；
- 当前阶段使用 mock 数据；
- 如果该论文暂无消息，则显示建议提问卡片；
- 对话区可支持滚动，但要固定输入框位置。

## 6.8 主功能页关键交互流程

### 场景一：初次进入主功能页

1. 默认选中第一篇论文；
2. 右侧显示该论文的概览；
3. 对话区显示该论文的历史消息或建议问题；
4. 用户可以立即开始查看或提问。

### 场景二：切换论文

1. 用户点击左侧另一篇论文；
2. 右侧标题、内容、标签和对话上下文全部切换；
3. 默认回到该论文的“概览”标签。

### 场景三：查看内容切换

1. 用户在右侧点击“原始摘要”“AI 总结”“翻译”“引文”等标签；
2. 当前标签内容变化；
3. 对话区保持可用；
4. 用户无需跳页面即可继续提问。

### 场景四：围绕单篇论文提问

1. 用户在当前论文下输入问题；
2. 消息流追加用户消息与系统回答；
3. 用户继续追问；
4. 形成围绕该论文的上下文链路。

## 9. 页面状态设计

第一阶段至少要覆盖以下状态：

- 有论文且默认选中
- 没有论文数据
- 搜索无结果
- 当前论文暂无总结
- 当前论文暂无翻译
- 当前论文暂无问答记录

这些状态不能等后面再补，否则页面只会有“理想状态”，不利于真实开发。

## 10. 第一阶段不做的内容

为了收敛范围，以下内容本阶段只预留入口，不做完整实现：

- 真实上传 PDF
- 后端问答接口
- 真正的总结生成逻辑
- 真正的翻译逻辑
- 多论文对比结果页
- 文献综述任务流
- 用户系统
- 权限控制

## 11. 第一阶段交付建议

前端第一阶段建议交付：

1. 欢迎页静态实现
2. 主功能页静态实现
3. 论文 mock 数据
4. 左侧论文切换右侧内容交互
5. 右侧内容标签切换交互
6. 单篇论文对话区 mock 交互

完成这 6 项后，就已经具备较完整的演示价值。

## 12. 建议开发顺序

建议按以下顺序推进：

1. 先实现欢迎页
2. 再实现主功能页整体布局
3. 接着实现左侧论文选择逻辑
4. 再做右侧标签切换
5. 然后补上对话区 mock 交互
6. 最后再接后端

这个顺序更稳，因为每一步都可视化、可验收、可演示。

## 13. 核心结论

当前最合适的前端方案不是做很多分散页面，而是做成：

- 一个用于建立产品认知的欢迎页；
- 一个承载核心使用流程的单页式主功能页。

在主功能页中：

- 左侧负责论文列表与选择；
- 右侧负责当前论文的内容与操作；
- 内容区通过标签切换展示摘要、总结、翻译、引文等信息；
- 对话区始终围绕当前论文存在。

这种结构最符合科研用户的实际使用习惯，也最适合在后端尚未完成时先做出高质量前端原型。
